# 开发过程思路与工作流说明

## 一、开发流程：Claude Code + DeepSeek V4 Pro 双模型协作

整个项目由 5 个对话驱动，每个对话承担不同的范式阶段：

           Phase 1    Phase 2   Phase 3+4

关键决策：

- **为什么分 5 个对话？** 单个对话上下文超过一定长度后，AI 会产生压缩导致的"遗忘"。把需求、设计、编码分到独立对话中，每个对话的上下文保持干净，决策质量更高。

## 二、典型问题与解决路径

### 问题 1：JDK 25 + Spring Boot 3.3 兼容性

**现象**：后端编译报 `找不到符号: 方法 setTitle(String)`，全部 13 个字段的 setter 都无法解析。

**诊断**：Lombok `@Data` 注解依赖 ByteBuddy 在编译期生成 getter/setter。JDK 25 中 `sun.misc.Unsafe` 被标记为 terminally deprecated，ByteBuddy 1.14.19 不支持 Java 25，导致注解处理器静默失败。

**解决路径**：
让ai更改lombok的版本,然后重新进行编译测试.

---

### 问题 2：Maven Wrapper 的跨平台陷阱

**现象**：`mvnw.cmd test` 在 PowerShell 中报 `No plugin found for prefix 'C'`。

**诊断**：`mvnw.cmd` 脚本将 `%MAVEN_CONFIG%`（值为 `C:\Users\Administrator\.m2`）作为 Maven 命令行参数传递。Maven 把 `C:` 解析为插件前缀，去远程仓库查找名为 `C` 的插件。

**解决路径**：
1. 创建 `mvnw.ps1`（PowerShell 原生脚本），直接用 `-classpath` + `MavenWrapperMain` 启动，不传 `MAVEN_CONFIG`
2. 在 README 中将所有 `mvn` 命令替换为 `.\mvnw.ps1`，代码块标记从 `bash` 改为 `powershell`
3. 去掉所有 `&&` 链式操作符（PowerShell 不支持），改为独立行

## 三、对"工程化 AI 开发"的理解

工程化ai开发,我认为是程序员通过prompt让ai进行流程化开发,然后程序员进行审核的一个过程.相较于传统的"古法编程",速度完全不在一个数量级上,之前需要很多人很长时间完成的项目,通过ai甚至可以一天完成,但同时也要对ai产生的代码进行审核检查,规范代码的开发的流程.
